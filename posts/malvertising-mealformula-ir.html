Hunting Through Chrome Incognito: How SRUM and Umbrella DNS Solved a Malvertising Case

2026-03-18 · ~15 min read
IR DFIR SRUM

Talos Threat Hunting flagged chrome.exe downloading and executing a suspicious binary on a Food Services Manager's workstation. Severity: CRITICAL. What followed was one of those investigations where every evidence source came back empty — until two artifacts cracked the whole thing wide open.

Names, hostnames, and identifiers have been sanitized.

The Alert

The Talos flag was straightforward: Chrome had downloaded a file called MealFormula.exe and executed it. On its face, that's bad. An unknown executable pulled down by a browser and immediately run? That's textbook malware delivery.

I pulled the file and sent it to Tria.ge for sandbox analysis. The results were... anticlimactic. MealFormula.exe was a renamed copy of the legitimate WinRAR installer. Valid EV code signing certificate from win.rar GmbH. Sandbox score 5/10 — all the flagged behaviors were normal installer stuff: writing to Program Files, creating registry keys, requesting elevation.

SHA-256: CFCA124C2141B4171CEBAD48CC286751910B408AEC69767DB2BC3AF38F277A06 — this is a clean, signed WinRAR installer. Not malware.

Severity downgraded. Looked like a false positive. A user downloaded WinRAR with a weird filename, AMP flagged it, case closed.

Except I couldn't figure out where it came from.

The Dead Ends

This is the part of the investigation that felt like hitting walls for hours. I needed to answer one question: how did MealFormula.exe get onto this machine?

Chrome browser history: 0 bytes. The History SQLite database existed but contained no records. I confirmed with Python:

import sqlite3
conn = sqlite3.connect('History')
cursor = conn.execute("SELECT url, title, last_visit_time FROM urls ORDER BY last_visit_time DESC LIMIT 20")
rows = cursor.fetchall()
print(f"Total rows: {len(rows)}")
# Total rows: 0

Zero rows. No downloads table entries either. The database was essentially freshly initialized.

Edge browser history: Nothing relevant. The user wasn't using Edge for this.

Zone.Identifier (Mark of the Web): No alternate data stream on the file. If it had been downloaded through a normal browser download, Windows should have stamped it with a Zone.Identifier ADS containing the referrer URL and download URL. Nothing.

USB history: Checked for removable media delivery. No evidence of any USB storage device.

Network shares: The file wasn't present on any mapped drives or accessible shares.

Every delivery vector eliminated. Chrome history empty. Edge clean. No MOTW. No USB. No network shares. The file appeared on the machine like it materialized out of thin air.

I was stuck. AMP trajectory confirmed Chrome downloaded the file, but Chrome's own records had been wiped clean. Something wasn't adding up.

SRUM: The Breakthrough

Here's where it got interesting. If Chrome's history was empty but AMP saw Chrome downloading the file, what if the user was running Chrome in Incognito mode?

Incognito doesn't write to the History database. It doesn't save downloads. It doesn't create cookies on disk. From a browser forensics perspective, it's a black hole. But Incognito can't hide from the operating system.

The System Resource Usage Monitor (SRUM) is a Windows telemetry database at C:\Windows\System32\sru\srudb.dat. It tracks per-application resource usage — CPU time, network bytes sent/received, foreground time — in time-bucketed records. It doesn't care if Chrome is in Incognito or not. It just sees chrome.exe moving bytes.

I pulled the SRUM database using VSS and parsed it two ways — first with SrumECmd for a quick look, then with Python and sqlite3 for custom queries against the parsed output.

# After parsing with SrumECmd, query the network usage data
import sqlite3, csv

# SrumECmd outputs to CSV - load and filter for chrome.exe
with open('SrumECmd_NetworkUsage_Output.csv', 'r') as f:
    reader = csv.DictReader(f)
    chrome_rows = [r for r in reader if 'chrome' in r['ExeInfo'].lower()]

for row in chrome_rows:
    print(f"{row['Timestamp']}  Sent: {row['BytesSent']}  Recv: {row['BytesRecvd']}")

The results told the whole story. Chrome had been running with heavy network usage despite the empty browser profile. The user had been running Chrome in Incognito mode daily since at least January 2026.

SRUM proved Chrome was being heavily used when every browser artifact said it wasn't. Incognito erases browser history, but it can't erase OS-level telemetry. The ghost had fingerprints.

But SRUM only tells you how much data moved, not where it went. I needed DNS.

Umbrella DNS: The Full Picture

We run Cisco Umbrella across the environment, which means every DNS query from every machine gets logged. Incognito mode doesn't encrypt your DNS queries (unless you're running DoH, which this machine wasn't). Every site the user visited in Incognito still generated a DNS lookup that Umbrella captured.

I pulled the Umbrella DNS logs for the workstation on March 13th. Here's the exact attack chain, reconstructed from DNS query timestamps:

Time (UTC)DomainCategorySignificance
12:11:04sallysbakingaddiction[.]comFood & RecipesLegitimate recipe blog — user was browsing recipes
12:11:05embed.filekitcdn[.]comCDNPayload delivery CDN — loaded by malicious ad creative
12:11:37meal-formula[.]comUncategorizedMalvertising landing page — redirect from ad network
12:11:44——MealFormula.exe download initiated

There it was. The user — a Food Services Manager — was reading recipes on a legitimate baking blog. Makes complete sense for their role. An ad served through the site's ad network (Adthrive/Raptive) contained a malicious creative that loaded content from embed.filekitcdn[.]com, which redirected the browser to meal-formula[.]com. That domain served up MealFormula.exe.

The entire attack took 40 seconds from initial ad load to payload download. The user didn't go looking for this file. They were reading a recipe and a malicious ad hijacked their browser.

Mystery solved. Delivery vector: malvertising via a legitimate recipe website. The ad network served a malicious creative that redirected to a payload download. Chrome Incognito hid it from browser forensics, but Umbrella DNS captured every step.

Campaign Attribution

The domain meal-formula[.]com is part of a documented malvertising campaign variously called FoodFormula or RecipeLister. The campaign specifically targets food service and healthcare workers — people who are likely to be browsing recipe sites, nutritional resources, and food ordering portals during their workday.

Related campaign infrastructure includes:

  • foodformulasetup[.]com — alternate landing page
  • recipelister[.]com — alternate landing page
  • sync.p7cloud[.]net — tracking/analytics beacon
  • embed.filekitcdn[.]com — payload CDN

The typical payload for this campaign is a Node.js-based infostealer — it drops a bundled Node.js runtime, extracts via 7-Zip, and runs a JavaScript payload that harvests browser credentials, cookies, and crypto wallet data. The characteristic artifacts are System.dll, a 7z-out directory, and Node.js processes spawning from unusual paths.

But here's the thing: this machine got the legitimate WinRAR installer instead of the infostealer. My theory is the campaign was in a setup or testing phase when this user hit it. Malvertising campaigns frequently rotate payloads, and serving a clean binary during initial deployment helps them avoid early detection while they test their redirect chain.

Full IOC Sweep

Even though the delivered payload was clean, I wasn't going to take that on faith. I ran a full sweep for every known indicator associated with the FoodFormula/RecipeLister campaign:

  • System.dll — not present anywhere on disk
  • Node.js — no Node.js runtime installed or extracted
  • 7z-out directory — not found
  • Persistence mechanisms — no new scheduled tasks, no Run/RunOnce registry keys, no startup folder entries
  • Campaign payload hash (B179BEC73EFE194BFFB69EEEDC8C136B5E7506A10B3589D223B1E6F8A8FBFB53) — not present
  • Network connections — no active connections to campaign infrastructure
All campaign indicators negative. No infostealer components, no persistence, no C2 communication. The machine was NOT compromised. The only artifact was the clean WinRAR installer sitting in the Downloads folder.

Registry Forensics

I used remote registry exports to nail down a few remaining questions.

BAM (Background Activity Moderator)

BAM tracks the last execution time of programs. It confirmed chrome.exe was last run on March 15th at 2:29 PM PDT — two days after the download. The user continued using Chrome in Incognito after the incident, which was consistent with their daily pattern.

MountedDevices and PortableDevices

I needed to definitively rule out USB as a delivery vector. HKLM\SYSTEM\MountedDevices showed only two entries:

  • C: — the NVMe system drive
  • D: — DVD drive

Zero USB storage devices had ever been connected to this machine. The PortableDevices registry key was completely empty. No phones, no thumb drives, no external hard drives. Ever.

Registry forensics is underrated for IR. BAM gives you program execution timestamps without needing prefetch or amcache. MountedDevices + PortableDevices definitively ruled out USB delivery in seconds. These are fast checks via remote registry export — no need to image the drive for quick answers.

Explaining the Upload Ratios

Remember those SRUM numbers that worried me? Chrome consistently sending nearly 2x what it received? That needed an explanation before I could close this out.

Umbrella DNS answered that too. Looking at the full browsing session for March 13th, the user's Incognito activity included:

  • Facebook — scrolling feed, watching videos, uploading photos
  • YouTube — video streaming (the upload is WebRTC/QUIC overhead)
  • Google account — personal email, Google Drive
  • Sysco food ordering portal — legitimate work task
  • Dayforce HCM — timekeeping system, also work-related

The high upload ratios were explained by video streaming protocols and social media activity, not data exfiltration. Facebook and YouTube video in particular generate significant upstream traffic for quality negotiation, analytics telemetry, and connection maintenance. Mix in photo uploads and Google Drive usage, and the numbers made perfect sense.

The user was doing personal browsing in Incognito to keep it off their work browser history — mixed in with some legitimate work tasks. That's a policy conversation, not a security incident.

Key Takeaways

  • SRUM defeats Incognito. Chrome leaves zero disk artifacts in Incognito mode, but SRUM still tracks per-process network bytes at the OS level. It proved Chrome was being actively used when every browser artifact said it wasn't. Always collect srudb.dat early — esentutl /y /vss to grab it while the file is locked.
  • Umbrella DNS is your browser history backup. Incognito can't hide from DNS. The full browsing session — including the exact attack chain — was reconstructed from DNS query logs alone. If you have network-level DNS logging, you effectively have a shadow browser history for every machine.
  • Malvertising targets specific industries. This campaign specifically targeted food service and healthcare workers through recipe and food-related sites. The ad network (Adthrive/Raptive) served the malicious creative. Legitimate sites can deliver malware through their ad supply chain.
  • Not every "known malware domain" means compromise. meal-formula[.]com is part of a documented infostealer campaign, but the actual payload delivered to this machine was a clean WinRAR installer. Don't panic — investigate. The domain hit alone would have justified a full wipe if we hadn't dug deeper.
  • Image first, ask questions later. On a related case around the same time, the machine was wiped before full evidence was collected. Lesson learned the hard way. Get your forensic image before remediation starts, even if the case looks like a false positive.
  • Registry is underrated for IR. BAM gives you program execution timestamps. MountedDevices and PortableDevices definitively ruled out USB delivery. These are fast checks via remote registry export that can answer critical questions in minutes.
